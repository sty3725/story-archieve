# 인증 흐름

## 현재 목표

회원가입이 성공 응답을 반환한 상태에서, 실제 DB 저장과 로그인·대시보드 흐름을 완성한다.

## 회원가입 흐름

```text
회원가입 폼
→ POST 회원가입 API
→ 이메일 형식 검증
→ 중복 이메일 확인
→ 비밀번호 해시
→ 사용자 저장
→ 생성된 사용자 정보 반환
```

## 로그인 흐름

```text
로그인 폼
→ POST 로그인 API
→ 이메일로 사용자 조회
→ 비밀번호 해시 검증
→ 인증 정보 발급
→ 브라우저에 저장
→ 대시보드 이동
```

## 인증 저장 방식

실제 저장소 구현을 확인한 후 기존 방식을 유지한다.

가능한 방식:

- HttpOnly 쿠키
- Authorization Bearer 토큰
- 서버 세션

### 우선 원칙

- 기존 코드에 방식이 있으면 그 방식을 유지한다.
- 새로 정해야 한다면 보안성과 현재 구조를 비교한다.
- `localStorage`에 장기 토큰을 저장하는 방식은 신중하게 사용한다.
- 비밀키, 토큰 만료시간, 알고리즘은 환경변수로 관리한다.

## 보호 페이지 흐름

```text
사용자가 dashboard.html 접근
→ 프론트엔드가 인증 상태 확인
→ 현재 사용자 API 호출
→ 성공: 화면 표시
→ 401: 로그인 페이지 이동
```

프론트엔드 리다이렉트만으로 보호하지 않는다. 백엔드 API도 반드시 인증을 검증한다.

## 로그아웃 흐름

- 쿠키 방식: 서버 또는 브라우저에서 쿠키 무효화
- Bearer 토큰 방식: 브라우저 저장 토큰 삭제
- 대시보드와 보호 페이지에서 로그인 화면으로 이동
- 로그아웃 후 보호 API가 401을 반환하는지 확인

## 권장 엔드포인트

실제 코드와 충돌하지 않는 범위에서 확인 또는 구현한다.

```text
POST /auth/register
POST /auth/login
POST /auth/logout
GET  /auth/me
```

## 완료 조건

- 중복 이메일 가입이 차단된다.
- 비밀번호가 해시로 저장된다.
- 올바른 계정 로그인은 성공한다.
- 잘못된 비밀번호는 401을 반환한다.
- 존재하지 않는 사용자는 인증되지 않는다.
- 로그인 성공 후 대시보드로 이동한다.
- 인증 없이 `/auth/me`와 보호 API에 접근할 수 없다.
- 로그아웃 후 보호 페이지 접근이 차단된다.
