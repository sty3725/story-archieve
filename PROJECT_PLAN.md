# Writer Platform 프로젝트 기획 정리

이 문서는 프로젝트 환경 세팅 문서와 별도로, **프로젝트의 목적, 기능 우선순위, 개발 순서, 원고 관리 구조, 주요 화면 설계, DB 초안**을 정리한 기획/설계 문서다.

환경 설치, 가상환경, SFTP, SSH, FastAPI 실행 명령어 등은 `PROJECT_SETUP_GUIDE.md`에서 관리하고, 이 문서는 개발 방향을 잃지 않기 위한 기준서로 사용한다.

---

## 1. 프로젝트 목표와 기술 스택 결정 이유

### 1-1. 프로젝트 목표

이 프로젝트의 목표는 단순한 포트폴리오 소개 페이지가 아니라, **작가와 지망생이 자신의 원고를 구조화해서 관리하고, 작성하고, 버전 기록을 남길 수 있는 웹 기반 원고 관리 플랫폼**을 만드는 것이다.

핵심 컨셉은 다음과 같다.

```text
작가가 자기 원고를 Git처럼 구조화하고,
버전 관리하고,
에디터에서 바로 작성하고,
필요하면 피드백까지 받을 수 있는 웹 플랫폼
```

초기 버전에서는 모든 기능을 완성하려 하지 않고, 포트폴리오로 보여줄 수 있을 만큼의 핵심 흐름을 우선 구현한다.

초기 핵심 흐름은 다음과 같다.

```text
회원가입 / 로그인
↓
내 작품 목록 확인
↓
작품 내부 구조 확인
↓
문서 선택 및 미리보기
↓
문서 작성 / 저장
↓
버전 기록 확인
```

### 1-2. 기술 스택

최종적으로 선택한 기본 기술 스택은 다음과 같다.

```text
Frontend: HTML / CSS / JavaScript
Backend: Python FastAPI
Database: PostgreSQL
API 통신: fetch API
개발 환경: VS Code
배포/원격 개발 보조: SFTP, SSH, GitHub
```

### 1-3. FastAPI를 선택한 이유

FastAPI를 선택한 이유는 다음과 같다.

```text
1. Python 기반이라 학습과 구현 진입장벽이 낮다.
2. REST API 서버를 만들기 좋다.
3. 프론트엔드와 fetch API로 연결하기 쉽다.
4. 자동 API 문서 기능이 있어 테스트와 설명이 편하다.
5. 포트폴리오에서 백엔드 API 설계 경험으로 설명하기 좋다.
```

한국 취업 시장에서 FastAPI가 Django나 Spring만큼 범용적인 주력 스택은 아니지만, 이 프로젝트에서는 다음과 같이 설명할 수 있다.

```text
FastAPI 기반 REST API 서버를 직접 설계하고,
프론트엔드와 fetch API로 연동했으며,
PostgreSQL을 사용해 사용자, 원고, 문서, 버전 데이터를 관리했다.
```

즉, 중요한 것은 FastAPI 이름 자체보다 **API 서버 설계, DB 연동, 프론트-백엔드 통신, 실제 기능 구현 경험**을 보여주는 것이다.

### 1-4. PostgreSQL을 선택한 이유

PostgreSQL을 사용하는 이유는 다음과 같다.

```text
1. 실제 서비스에서 많이 쓰이는 관계형 데이터베이스다.
2. 사용자, 작품, 문서, 버전처럼 관계가 명확한 데이터를 관리하기 좋다.
3. 포트폴리오에서 MySQL 계열과 함께 충분히 설명 가능한 DB 경험이 된다.
4. 트리 구조, 정렬 순서, 관계형 테이블 설계를 연습하기 좋다.
```

## 2. 기능 우선순위

이 프로젝트는 기능 범위가 커질 수 있기 때문에 처음부터 모든 기능을 구현하려 하면 완성 가능성이 떨어진다. 따라서 기능 우선순위를 명확히 둔다.

### 2-1. 전체 기능 우선순위

```text
0. 기본 UI / 회원가입 / 로그인
1. 원고 관리
2. 에디터
3. 버전 관리
4. 댓글 / 피드백
5. 라벨
6. 브랜치
7. 버전 비교
```

### 2-2. 0순위: 기본 UI / 회원가입 / 로그인

가장 먼저 필요한 것은 사용자가 서비스에 들어와서 자신의 데이터를 구분할 수 있는 최소 구조다.

포함 기능:

```text
- 기본 레이아웃
- 회원가입 화면
- 로그인 화면
- 로그아웃
- 로그인 상태에 따른 화면 분기
```

초기에는 보안 고도화보다 기능 흐름 확인이 우선이다.

### 2-3. 1순위: 원고 관리

이 프로젝트의 핵심 기능이다.

포함 기능:

```text
- 작품 생성
- 작품 목록 조회
- 작품 선택
- 작품 내부 구조 조회
- 중간 카테고리 생성
- 문서 생성
- 문서 미리보기
- 구조 변경 모드
```

사용자가 자기 원고를 파일 시스템처럼 정리할 수 있어야 한다.

### 2-4. 2순위: 에디터

초기 에디터는 고급 문서 편집기가 아니어도 된다.

초기 구현:

```text
- textarea 기반 작성 영역
- 문서 제목 표시
- 저장 버튼
- 마지막 저장 시간 표시
```

이후 필요하면 TipTap, Quill, CodeMirror 같은 라이브러리를 검토한다.

### 2-5. 3순위: 버전 관리

원고 관리 플랫폼의 차별점이 될 수 있는 기능이다.

초기 구현:

```text
- 저장 시 버전 기록 생성
- 버전 목록 조회
- 특정 버전 내용 확인
```

고급 기능인 비교(diff), 브랜치, 병합은 후순위로 둔다.

### 2-6. 후순위 기능

댓글, 피드백, 라벨, 브랜치, 버전 비교는 프로젝트가 1차 완성된 뒤 확장 기능으로 다룬다.

후순위 기능:

```text
- 댓글 / 에디터 피드백
- 원고 라벨
- 브랜치
- 버전 비교
- 협업 기능
- 클래스룸 / 강의실 기능
```

## 3. 개발 순서

개발은 기능 목록 순서와 완전히 같지 않다. 먼저 실행 가능한 환경을 만들고, 화면을 만들고, 더미 데이터로 동작을 확인한 뒤, 백엔드와 DB를 연결한다.

### 3-1. 전체 개발 순서

```text
0. FastAPI 기반 실행 확인 환경 만들기
1. 프론트엔드 기본 레이아웃 만들기
2. Dashboard 정적 화면 만들기
3. Archive / Manuscripts 정적 화면 만들기
4. Editor 정적 화면 만들기
5. 더미 데이터로 화면 동작 구현
6. FastAPI 라우터 분리
7. 원고 관리 API 만들기
8. PostgreSQL 연결
9. 프론트 fetch API 연결
10. 로그인 / 회원가입 연결
11. 버전 저장 기능 추가
12. 에디터 고도화
```

### 3-2. 0단계: 실행 확인 환경

프론트엔드를 단순히 HTML 파일 더블클릭으로 확인하지 않고, FastAPI 서버 위에서 확인한다.

목표:

```text
FastAPI 실행
↓
브라우저에서 http://127.0.0.1:8000 접속
↓
frontend/index.html 표시
↓
CSS / JS 정상 로드
↓
/api/health 호출 확인
```

이 단계가 필요한 이유:

```text
- 실제 서비스 구조와 비슷한 방식으로 개발할 수 있다.
- CSS, JS, API 경로 문제를 초기에 잡을 수 있다.
- 이후 프론트와 백엔드 연결 테스트가 쉬워진다.
```

### 3-3. 1단계: 공통 레이아웃

가장 먼저 공통 레이아웃을 잡는다.

포함 요소:

```text
- 전체 앱 컨테이너
- 상단 헤더
- 좌측 사이드바
- 메인 콘텐츠 영역
- 공통 카드 UI
- 공통 버튼
- 공통 입력창
- 공통 모달 스타일
```

이 단계에서는 실제 데이터보다 **서비스처럼 보이는 기본 뼈대**가 중요하다.

### 3-4. 2~4단계: 주요 페이지 정적 구현

먼저 정적 화면으로 각 페이지를 만든다.

대상 페이지:

```text
- dashboard.html
- manuscripts.html
- editor.html
```

정적 화면을 먼저 만드는 이유:

```text
- API 없이도 화면 구조를 검증할 수 있다.
- 어떤 데이터가 필요한지 자연스럽게 드러난다.
- DB 설계를 화면 기준으로 더 정확하게 잡을 수 있다.
```

### 3-5. 5단계: 더미 데이터로 동작 구현

백엔드 연결 전에 JS 더미 데이터로 화면을 움직이게 만든다.

구현할 동작:

```text
- 작품 클릭
- 작품 내부 구조 표시
- 문서 클릭
- 미리보기 변경
- 트리 펼치기 / 접기
- 변경 모드 on/off
```

이 단계가 필요한 이유:

```text
- UX가 실제로 자연스러운지 확인할 수 있다.
- 백엔드 구현 전에 필요한 API 형태를 예측할 수 있다.
```

### 3-6. 6단계 이후: API와 DB 연결

화면과 더미 동작이 어느 정도 잡히면 FastAPI 라우터와 DB를 연결한다.

진행 순서:

```text
1. routers/manuscripts.py 생성
2. GET /api/manuscripts 구현
3. GET /api/manuscripts/{id}/nodes 구현
4. POST /api/manuscripts 구현
5. POST /api/nodes 구현
6. PostgreSQL 테이블 연결
7. 프론트 fetch API 연결
```

## 4. 원고 구조 설계 원칙

이 프로젝트의 핵심은 원고를 단순한 문서 목록으로 관리하지 않는 것이다. 사용자가 작품마다 자유로운 구조를 만들 수 있어야 한다.

### 4-1. 기본 구조

기본 원칙은 다음과 같다.

```text
작품 → 중간 카테고리 → 텍스트 최소 단위
```

예시 1:

```text
작품
 ├─ 1부
 │  ├─ 1장
 │  │  └─ 본문 문서
 │  └─ 2장
 └─ 설정집
    ├─ 인물
    └─ 세계관
```

예시 2:

```text
작품
 ├─ 시즌 1
 │  ├─ 1화
 │  ├─ 2화
 │  └─ 3화
 └─ 외전
```

예시 3:

```text
작품
 ├─ 본편
 │  ├─ 001.txt
 │  ├─ 002.txt
 │  └─ 003.txt
 ├─ 캐릭터 노트
 └─ 플롯 메모
```

### 4-2. 고정 구조를 피하는 이유

처음부터 다음과 같이 고정하면 자유도가 떨어진다.

```text
작품 → 챕터 → 회차
```

작품마다 구조가 다를 수 있다.

```text
- 웹소설은 회차 중심일 수 있다.
- 장편 소설은 부/장 중심일 수 있다.
- 설정집은 인물/지역/연표 중심일 수 있다.
- 습작은 폴더 없이 문서만 있을 수 있다.
```

따라서 구조를 고정하지 않고, 파일 시스템에 가까운 트리 구조로 설계한다.

### 4-3. 노드 기반 구조

작품 내부의 모든 단위는 `node`로 다룬다.

노드 타입 예시:

```text
folder: 중간 카테고리
chapter: 장/회차 같은 구조 단위
document: 실제 본문이 있는 문서
note: 설정 메모 또는 보조 문서
```

초기에는 타입을 단순하게 시작해도 된다.

```text
folder
document
```

### 4-4. parent_id 구조

트리 구조는 `parent_id`로 표현한다.

```text
id: 현재 노드 ID
parent_id: 부모 노드 ID
```

최상위 노드는 `parent_id`가 비어 있거나 NULL이다.

예시:

```text
작품 A
 ├─ 1부
 │  └─ 1화
 └─ 설정집
```

테이블로 표현하면:

```text
id | parent_id | title
1  | null      | 1부
2  | 1         | 1화
3  | null      | 설정집
```

## 5. Archive Page UX 원칙

Archive Page는 이 프로젝트의 핵심 화면이다. 단순한 목록 페이지가 아니라, 사용자가 자기 작품 구조를 확인하고 안전하게 관리하는 화면이다.

### 5-1. Archive Page의 역할

Archive Page의 목적:

```text
- 작품 목록을 보여준다.
- 선택한 작품의 내부 구조를 보여준다.
- 문서를 클릭하면 내용을 미리보기로 보여준다.
- 구조 변경은 별도 변경 모드에서만 가능하게 한다.
```

### 5-2. 기본 레이아웃

권장 레이아웃:

```text
┌─────────────────────────────────────┐
│ Archive                             │
├──────────────┬──────────────┬───────┤
│ 작품 목록    │ 구조 트리      │ 미리보기 │
│              │              │       │
└──────────────┴──────────────┴───────┘
```

영역별 역할:

```text
작품 목록: 사용자의 작품들을 보여준다.
구조 트리: 선택한 작품 내부의 폴더/문서 구조를 보여준다.
미리보기: 선택한 문서의 내용을 읽기 전용으로 보여준다.
```

### 5-3. 기본 모드와 변경 모드 분리

중요 UX 원칙:

```text
기본 상태에서는 읽기 / 탐색 / 미리보기만 가능
변경 모드에 들어가야 추가 / 삭제 / 이동 / 이름 변경 가능
```

이렇게 하는 이유:

```text
- 의도치 않은 삭제를 막을 수 있다.
- 사용자가 구조를 탐색할 때 부담이 줄어든다.
- 포트폴리오에서 UX 안전성을 설명하기 좋다.
```

### 5-4. 기본 모드에서 가능한 것

```text
- 작품 선택
- 폴더 펼치기 / 접기
- 문서 선택
- 문서 미리보기
- 에디터로 이동
```

### 5-5. 변경 모드에서 가능한 것

```text
- 작품 이름 변경
- 폴더 추가
- 문서 추가
- 노드 이름 변경
- 노드 삭제
- 노드 이동
- 정렬 변경
```

초기 버전에서는 모든 변경 기능을 다 넣지 않아도 된다.

초기 변경 모드 최소 기능:

```text
- 폴더 추가
- 문서 추가
- 이름 변경
- 삭제
```

### 5-6. 삭제 UX

삭제는 반드시 확인 절차를 둔다.

권장 방식:

```text
1. 삭제 버튼 클릭
2. 삭제 대상 이름 표시
3. 확인 모달 표시
4. 확인 후 삭제
```

문서 미리보기를 지원하는 이유도 잘못된 문서를 삭제하지 않게 돕기 위한 것이다.

## 6. DB 테이블 초안

초기 DB는 너무 복잡하게 시작하지 않는다. 기능 구현에 필요한 최소 테이블부터 시작하고, 이후 확장한다.

### 6-1. 초기 핵심 테이블

```text
users
manuscripts
manuscript_nodes
manuscript_versions
```

### 6-2. users 테이블

사용자 정보를 저장한다.

```text
users
- id
- email
- password_hash
- nickname
- created_at
- updated_at
```

역할:

```text
- 로그인 사용자 구분
- 사용자의 작품 소유권 구분
```

### 6-3. manuscripts 테이블

작품 단위 정보를 저장한다.

```text
manuscripts
- id
- user_id
- title
- description
- created_at
- updated_at
```

관계:

```text
users 1명 → manuscripts 여러 개
```

`user_id`는 `users.id`를 참조한다.

### 6-4. manuscript_nodes 테이블

작품 내부 구조를 저장하는 핵심 테이블이다.

```text
manuscript_nodes
- id
- manuscript_id
- parent_id
- title
- type
- content
- sort_order
- created_at
- updated_at
```

역할:

```text
- 작품 내부의 폴더, 장, 회차, 문서 구조 표현
- parent_id를 통해 트리 구조 구성
- document 타입일 경우 content에 본문 저장 가능
```

관계:

```text
manuscripts 1개 → manuscript_nodes 여러 개
manuscript_nodes 1개 → 하위 manuscript_nodes 여러 개
```

주의:

```text
- parent_id는 같은 테이블의 id를 참조한다.
- 최상위 노드는 parent_id가 NULL이다.
- sort_order로 같은 부모 아래의 순서를 관리한다.
```

### 6-5. manuscript_versions 테이블

문서 저장 기록을 저장한다.

```text
manuscript_versions
- id
- node_id
- title
- content
- version_number
- memo
- created_at
```

역할:

```text
- 문서 저장 시점의 내용을 보존
- 이전 버전 확인
- 나중에 버전 비교 기능의 기반이 됨
```

관계:

```text
manuscript_nodes 1개 → manuscript_versions 여러 개
```

### 6-6. 추후 확장 테이블

후순위 기능이 들어가면 다음 테이블을 추가할 수 있다.

```text
comments
labels
node_labels
branches
classrooms
feedbacks
```

초기에는 만들지 않는다. 이유는 다음과 같다.

```text
- 테이블이 많아지면 구현 부담이 커진다.
- 핵심 원고 관리 흐름을 먼저 완성하는 것이 중요하다.
- 실제 화면과 API가 잡힌 뒤 확장하는 편이 안전하다.
```

## 7. 프론트엔드 레이아웃 개발 순서

프론트엔드는 처음부터 API와 DB를 붙이지 않고, 정적 레이아웃과 더미 데이터를 통해 화면 완성도를 먼저 확인한다.

### 7-1. 최초 작업 파일

가장 먼저 손댈 파일:

```text
frontend/index.html
frontend/css/common.css
frontend/js/main.js
```

이후 추가 작업 파일:

```text
frontend/pages/dashboard.html
frontend/pages/manuscripts.html
frontend/pages/editor.html
frontend/css/dashboard.css
frontend/css/manuscripts.css
frontend/css/editor.css
frontend/js/dashboard/dashboard.js
frontend/js/manuscripts/manuscripts.js
frontend/js/editor/editor.js
```

### 7-2. 공통 레이아웃 구성

먼저 사이트 전체에 공통으로 적용될 레이아웃을 만든다.

구성:

```text
- app shell
- header
- sidebar
- main content
- page title area
- card component
- button style
- form input style
```

권장 구조:

```text
<body>
  <div class="app-shell">
    <aside class="sidebar"></aside>
    <div class="app-main-area">
      <header class="app-header"></header>
      <main class="page-content"></main>
    </div>
  </div>
</body>
```

### 7-3. 공통 디자인 방향

디자인 방향:

```text
- 블랙 계열 배경
- 흰색 또는 푸른빛이 섞인 얇은 테두리
- glassmorphism 패널
- backdrop-filter blur
- 카드형 정보 배치
- 실제 서비스처럼 읽기 쉬운 UI
```

기존 포트폴리오에서 사용한 미래적인 HUD 느낌을 일부 가져오되, 이 프로젝트에서는 실사용성이 중요하다.

비율로 보면 다음 정도가 적절하다.

```text
화려한 비주얼 30%
실사용 가능한 관리 UI 70%
```

### 7-4. Dashboard Page 개발

Dashboard는 첫 화면이다.

포함 요소:

```text
- 최근 작업한 원고
- 전체 작품 수
- 최근 수정일
- 빠른 이동 버튼
- 간단한 상태 카드
```

초기에는 더미 데이터로 카드만 배치한다.

### 7-5. Manuscripts / Archive Page 개발

가장 중요한 화면이다.

개발 순서:

```text
1. 3분할 레이아웃 생성
2. 작품 목록 영역 생성
3. 구조 트리 영역 생성
4. 미리보기 영역 생성
5. 더미 작품 데이터 연결
6. 작품 클릭 시 구조 변경
7. 문서 클릭 시 미리보기 변경
8. 변경 모드 버튼 추가
9. 변경 모드일 때만 추가/삭제 버튼 표시
```

이 화면은 백엔드 연결 전에 JS 더미 데이터로 충분히 테스트해야 한다.

### 7-6. Editor Page 개발

초기 에디터는 단순하게 시작한다.

구성:

```text
- 상단 문서 정보 바
- 좌측 문서 목록
- 중앙 textarea 작성 영역
- 우측 메모 / 버전 정보 패널
- 저장 버튼
```

초기에는 다음 기능만 구현한다.

```text
- 문서 제목 표시
- textarea에 내용 표시
- 저장 버튼 클릭 이벤트
- 저장 완료 메시지 표시
```

### 7-7. 더미 데이터 단계

초기 더미 데이터 예시:

```javascript
const manuscripts = [
  {
    id: 1,
    title: "검은 도시의 기록",
    description: "도시 판타지 장편 프로젝트",
    nodes: [
      {
        id: 101,
        parentId: null,
        title: "1부",
        type: "folder"
      },
      {
        id: 102,
        parentId: 101,
        title: "1화 - 시작",
        type: "document",
        content: "첫 문장 테스트..."
      }
    ]
  }
];
```

더미 데이터로 확인할 것:

```text
- 화면 배치가 자연스러운가?
- 작품/문서 선택 흐름이 이해되는가?
- 미리보기 영역이 충분한가?
- 변경 모드가 직관적인가?
- API로 바꿔도 무리 없는 데이터 구조인가?
```

### 7-8. API 연결 전 점검

API 연결 전에 다음이 되어 있어야 한다.

```text
- 정적 레이아웃이 깨지지 않는다.
- 더미 데이터로 작품 목록이 출력된다.
- 더미 데이터로 문서 트리가 출력된다.
- 문서를 클릭하면 미리보기가 바뀐다.
- 변경 모드 UI가 구분된다.
```

이 상태가 되면 FastAPI와 PostgreSQL 연결로 넘어간다.

## 마무리 기준

이 문서 기준으로 1차 목표는 다음과 같다.

```text
FastAPI 서버 위에서 프론트엔드가 실행되고,
Dashboard / Archive / Editor의 기본 화면이 있으며,
더미 데이터로 원고 구조를 탐색할 수 있고,
이후 API와 DB로 자연스럽게 연결 가능한 상태
```

이 상태까지 도달하면 포트폴리오 프로젝트의 1차 골격이 완성된 것으로 본다.
