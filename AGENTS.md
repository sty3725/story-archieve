# Story Archive — Codex 작업 지침

이 문서는 Codex가 이 저장소에서 작업할 때 따라야 하는 최상위 지침이다.

## 1. 프로젝트 목적

Story Archive는 작가가 원고를 작성·정리·버전 관리하고, 최근 문서와 작업 현황을 대시보드에서 확인하며, 강의실 기능까지 사용할 수 있도록 만드는 웹 애플리케이션이다.

현재 우선순위는 **회원가입 → 로그인 → 인증된 대시보드 진입** 흐름을 안정적으로 완성하는 것이다.

## 2. 기술 스택

- Backend: Python, FastAPI
- Database: PostgreSQL
- ORM: 현재 저장소 구현을 우선 확인
- Frontend: HTML, CSS, Vanilla JavaScript
- Authentication: 저장소 구현 확인 후 JWT 또는 세션 방식 중 기존 방식을 유지
- Version control: Git / GitHub

## 3. 작업 전 필수 절차

1. 이 파일과 `docs/`의 관련 문서를 먼저 읽는다.
2. 실제 저장소 구조와 문서의 예상 구조가 일치하는지 확인한다.
3. 문서보다 현재 코드가 우선이다. 불일치가 있으면 임의로 덮어쓰지 말고 작업 결과에 기록한다.
4. 수정하기 전에 관련 라우터, 모델, 스키마, 서비스, 프론트엔드 호출부를 함께 추적한다.
5. 기존 구현을 최대한 재사용하고 변경 범위를 최소화한다.
6. 추측으로 새 프레임워크나 대형 의존성을 추가하지 않는다.

## 4. 코드 수정 원칙

- 프론트엔드 프레임워크를 새로 도입하지 않는다.
- 백엔드 디렉터리 구조를 임의로 재편하지 않는다.
- 기존 API 응답 형식과 에러 처리 방식을 유지한다.
- 비밀번호는 평문 저장하지 않는다.
- 인증이 필요한 API에는 일관된 인증 검증을 적용한다.
- 사용자 소유 데이터는 다른 사용자와 분리한다.
- 모델을 수정하면 관련 스키마, 서비스, 라우터, 마이그레이션 영향을 함께 점검한다.
- 환경변수와 비밀키를 코드에 하드코딩하지 않는다.
- 파일명을 바꾸거나 삭제해야 한다면 먼저 참조 위치를 전부 찾는다.

## 5. 검증 원칙

작업 후 가능한 범위에서 다음을 확인한다.

- 서버가 정상 기동되는가
- import 오류가 없는가
- 회원가입이 정상 작동하는가
- 로그인 성공/실패 응답이 적절한가
- 인증 없이 보호 API에 접근할 수 없는가
- 로그인 후 대시보드로 이동하는가
- 브라우저 새로고침 후 인증 상태가 예상대로 유지되는가
- 로그아웃 후 보호 페이지 접근이 차단되는가
- 사용자 A의 데이터가 사용자 B에게 노출되지 않는가

## 6. 작업 결과 보고 형식

작업 완료 후 아래 형식으로 요약한다.

1. 수정한 파일
2. 구현한 내용
3. 실행 및 검증 방법
4. 검증 결과
5. 남은 문제 또는 위험
6. 다음 권장 작업

## 7. 현재 최우선 작업

`docs/DEVELOPMENT_STATUS.md`와 `docs/ROADMAP.md`를 기준으로 한다.

첫 번째 목표:

- 로그인 API 확인 또는 구현
- 프론트엔드 로그인 폼 연결
- 로그인 성공 후 대시보드 이동
- 대시보드 API 보호
- 현재 로그인 사용자 조회
- 로그아웃 처리
